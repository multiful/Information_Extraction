# KG 적재 단계: 저confidence 트리플 LLM 보정 계획 (미착수, 설계만)

> **최종 업데이트**: 2026-07-14: 최초 작성. 모델(dk_gat/atlop/트랙1 중 최종 채택 모델)이 확정된
> 이후 착수 예정 — 지금은 설계만 해두고 구현하지 않음. OpenAI API 키는 이 문서/코드 어디에도
> 저장하지 않음(대화창에 노출된 키는 폐기 후 재발급 권장, `.env`는 `.gitignore`에 추가 완료).

## 목적 및 적용 범위

**언제**: 최종 모델이 확정된 뒤, KG(Knowledge Graph) 적재 단계에서만 사용.
**어디에**: ① `train_distant`에서 KG용으로 샘플링한 문서, ② 라벨 없는 `test` 데이터셋의 관계 —
이 둘에서 모델이 뽑은 트리플 중 **confidence가 애매한 것만** LLM으로 재검증.

**적용 안 하는 곳** (명확히): `train_annotated`/`dev`처럼 사람이 검증한 gold 라벨은 그대로 사용
(LLM 개입 불필요). **팀 모델 비교표(dev F1/Ign F1)에도 절대 섞지 않음** — PRD.md의 "동일 스코어러로
순수 모델만 비교"라는 규칙이 깨지므로, 이 보정은 KG 최종 출력물에만 적용되는 별도 후처리 레이어.

## 보안 (필수, 구현 시작 전에 반드시 지킬 것)

- API 키는 **환경변수**(`OPENAI_API_KEY`)로만 주입. 코드/커밋/로그 어디에도 하드코딩 금지.
- 실제 구현 시 로컬 `.env` 파일 사용 권장 (`.gitignore`에 이미 등록됨: `.env`, `.env.*`, `*.key`).
- 혹시라도 커밋 직전에 `git diff`/`git status`로 `.env`나 키 값이 포함 안 됐는지 한 번 더 확인.

## 가드레일 (비용 통제, 사용자 제시안 반영) A/B테스트 필요 성능,지연 트레이드오프 최적화

1. **Confidence margin 필터**: `|predicted_logit - TH_logit| < margin`(예: 0.5) 인 것만 대상.
   압도적으로 확신하는 예측은 건드리지 않음 — 애매한 경계선 케이스만 재검증.
2. **문서 단위 배칭**: 한 문서의 애매한 후보 여러 개를 **한 API 콜에 몰아서** 문서 문맥(가장 큰
   토큰 비중)을 한 번만 보냄. per-triple 콜 대비 문맥 토큰 중복 제거.
3. **콜 수 상한**: run당 최대 N콜 (예: 초기엔 N=500 정도로 시작해 실제 비용 보고 조정).
4. **저가 모델 사용**: gpt-4o-mini급으로 충분 — 개방형 생성이 아니라 "이 트리플이 맞는지/어느
   관계가 맞는지"를 고르는 좁은 판별 작업이라 고가 모델이 필요 없음.

## 추가 비용 정제 아이디어 (이번에 보강) A/B테스트 필요 성능,지연 트레이드오프 최적화

5. **출력 형식 제약(structured output/function calling)**: 자유 서술형 답변 대신 "이 96개 관계
   중 하나 고르기" 또는 "yes/no"만 강제 — 출력 토큰을 몇 개로 줄이고, 파싱 실패로 인한 재시도
   비용도 제거.
6. **2단계 confidence 밴드**: "매우 애매함"(margin ~0에 근접)만 풀 재분류(96개 중 재선택), "약간
   애매함"(margin이 더 크지만 threshold 안)은 훨씬 짧은 프롬프트로 "이 예측 맞아?"만 yes/no 확인
   — 후자가 더 많을 텐데 훨씬 저렴하게 처리.
7. **엔티티 타입 사전 필터(무료)**: LLM 호출 전에 규칙 기반으로 걸러내기 — 예를 들어 관계가
   "date of birth"인데 tail 엔티티 타입이 TIME이 아니면 애초에 말이 안 되는 후보라 LLM 호출 없이
   즉시 기각. 이러면 애매한 후보 중 일부는 공짜로 정리됨.
8. **캐싱**: (문서 제목, head, tail, 후보 관계) 키로 응답을 캐싱 — KG 재적재나 파이프라인 반복
   실행 시 같은 트리플을 또 API에 물어보지 않음.
9. **동일 사실 재사용**: 같은 (엔티티 이름, 엔티티 이름, 관계) 조합이 코퍼스 다른 곳에서 이미
   고confidence로 확인됐다면, 애매한 인스턴스는 API 호출 없이 그 확정 사실을 재사용 (이미
   `Scripts/docred_scorer.py`가 하는 fact-lookup 방식과 같은 아이디어).
10. **파일럿 먼저**: 전체 적용 전에 소규모 샘플(예: distant 500문서)로 먼저 돌려서 "LLM이 실제로
    모델 예측을 얼마나 자주 바꾸는지"(정정률) 확인. 정정률이 낮으면(모델과 거의 항상 일치) 굳이
    대규모로 확장할 실익이 적다는 뜻이므로, 여기서 진행 여부를 다시 판단.

## 대략적 비용 감 (참고용, 실측 아님)

- 문서당 예측 트리플 ~10~12개(dev 실측 기준) 중 애매한 것만 추리면 문서당 2~4개 수준으로 예상.
- 문서 단위 배칭 시 문서 수만큼만 콜 발생 — dev/test 규모(~1,000개)면 최대 1,000콜 안팎, 저가
  모델 기준 실제 비용은 크지 않을 것으로 예상.
- **단, train_distant(101,873개) 전체나 실제 프로덕션 코퍼스로 확장하면 후보 규모가 수십~수백
  배로 뛸 수 있음** — 반드시 9번(파일럿)으로 먼저 확인 후 규모를 정할 것.
